# Contributing

Thank you for considering a contribution to Data Engineering From Zero To Impact.

This repository is intended to be a practical learning system for modern Data Engineering. Contributions should help learners understand fundamentals, build real projects, improve stage by stage, and prepare for professional work.

## What Makes A Good Contribution

Good contributions are:

- Practical
- Accurate
- Well-structured
- Business-aware
- Beginner-conscious
- Production-minded
- Easy to follow
- Clear about assumptions and tradeoffs

The best contributions do more than explain a tool. They show how a concept or technique helps solve a real data problem.

## Contribution Areas

Useful contributions may include:

- Roadmap improvements
- Concept explanations
- Project labs
- Exercises
- Synthetic datasets
- Data dictionaries
- Pipeline examples
- SQL practice problems
- Python data engineering examples
- Data quality checks
- Architecture diagrams
- Portfolio case study templates
- Interview preparation material
- Corrections and clarifications

## Content Principles

Please follow the repository philosophy:

1. Fundamentals before tools.
2. Business problems before random tutorials.
3. Production thinking from the beginning.
4. Projects before theory-only learning.
5. Data quality, reliability, and trust.
6. AI-ready Data Engineering for the modern age.
7. Real-world business impact.

Avoid contributions that are only lists of tools, copied vendor descriptions, or high-level advice without practical application.

## Writing Guidelines

Use clear, practical, professional English.

Before submitting documentation, check that it answers:

- What is the learner expected to understand or build?
- Why does this topic matter in Data Engineering?
- What business problem does it connect to?
- What assumptions are being made?
- What are common mistakes?
- How can the learner validate their work?
- What would make the result portfolio-ready?

Prefer concrete examples over abstract claims.

## Project Lab Guidelines

Project labs should be complete enough for learners to use without guessing the intent.

A strong project lab should include:

- Business scenario
- Learning objectives
- Required prerequisites
- Source data description
- Target outcome
- Suggested architecture
- Step-by-step tasks
- Validation checks
- Expected outputs
- Documentation requirements
- Extension ideas
- Portfolio talking points

Projects should favor realistic domains such as retail, inventory, customer analytics, supplier performance, finance operations, small business reporting, support operations, and AI-ready knowledge bases.

## Data Guidelines

When contributing data:

- Use synthetic, public, or clearly licensed data.
- Do not include secrets, private credentials, personal data, or sensitive business records.
- Include a data dictionary when practical.
- Explain how the data was created or sourced.
- Identify known data quality issues if they are intentional.
- Keep datasets small enough for learners to run locally unless there is a clear reason otherwise.

## Code Guidelines

When contributing code:

- Keep examples readable.
- Include setup and run instructions.
- Use explicit configuration.
- Avoid unnecessary framework complexity.
- Handle errors where appropriate.
- Add validation checks for important data transformations.
- Add tests when logic affects correctness.
- Keep raw, processed, and output data separated.
- Do not commit generated files unless they are intentionally part of the learning material.

## Pull Request Checklist

Before opening a pull request:

- Read [README.md](README.md), [ROADMAP.md](ROADMAP.md), and [AGENTS.md](AGENTS.md).
- Confirm your change supports the repository mission.
- Check spelling, links, headings, and formatting.
- Make sure examples are practical and accurate.
- Remove secrets, private data, and unnecessary generated files.
- Add or update documentation for any new project or code.
- Update [CHANGELOG.md](CHANGELOG.md) for meaningful changes.

## Review Expectations

Maintainers may ask for changes if a contribution is:

- Too shallow
- Too tool-focused
- Missing business context
- Missing validation guidance
- Hard for beginners to follow
- Not aligned with the roadmap
- Duplicative with existing material
- Inaccurate or unsupported

Review is intended to keep the repository useful, coherent, and trustworthy.

## Suggested First Contributions

Good first contributions include:

- Fix unclear wording
- Add examples to an existing section
- Improve roadmap milestones
- Create a small SQL exercise
- Add a data dictionary template
- Draft a project brief for a realistic business scenario
- Add validation checks to an existing project
- Improve setup instructions

## Maintainer Notes

The repository should grow intentionally. New content should connect to the roadmap and help learners build toward professional competence.

When in doubt, choose clarity, usefulness, and trust over breadth.
